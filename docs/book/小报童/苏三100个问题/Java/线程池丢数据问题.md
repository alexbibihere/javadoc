# 线程池丢数据问题

## **<font style="color:rgb(34, 34, 34);background-color:rgb(248, 246, 244);">前言</font>**
<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">最近有位小伙伴在我的技术群里，问了我一个问题：服务down机了，线程池中如何保证不丢失数据？</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">这个问题挺有意思的，今天通过这篇文章，拿出来跟大家一起探讨一下。</font>

## **<font style="color:rgb(34, 34, 34);background-color:rgb(248, 246, 244);">1 什么是线程池？</font>**
<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">之前没有线程池的时候，我们在代码中，创建一个线程有两种方式：</font>

1. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">继承Thread类</font>
2. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">实现Runnable接口</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">虽说通过这两种方式创建一个线程，非常方便。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">但也带来了下面的问题：</font>

1. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">创建和销毁一个线程，都是比较耗时，频繁的创建和销毁线程，非常影响系统的性能。</font>
2. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">无限制的创建线程，会导致内存不足。</font>
3. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">有新任务过来时，必须要先创建好线程才能执行，不能直接复用线程。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">为了解决上面的这些问题，Java中引入了：线程池。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">它相当于一个存放线程的池子。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">使用线程池带来了下面3个好处：</font>

1. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">降低资源消耗。通过重复利用已创建的线程降低线程创建和销毁造成的消耗。</font>
2. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">提高响应速度。当任务到达时，可以直接使用已有空闲的线程，不需要的等到线程创建就能立即执行。</font>
3. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">提高线程的可管理性。线程是稀缺资源，如果无限制的创建，不仅会消耗系统资源，还会降低系统的稳定性。而如果我们使用线程池，可以对线程进行统一的分配、管理和监控。</font>

## **<font style="color:rgb(34, 34, 34);background-color:rgb(248, 246, 244);">2 线程池原理</font>**
<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">先看看线程池的构造器：</font>

```plain
public ThreadPoolExecutor(
    int corePoolSize,
    int maximumPoolSize,
    long keepAliveTime,
    TimeUnit unit,
    BlockingQueue workQueue,
    ThreadFactory threadFactory,
    RejectedExecutionHandler handler)
```

+ <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">corePoolSize：核心线程数，线程池维护的最少线程数。</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">maximumPoolSize：最大线程数，线程池允许创建的最大线程数。</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">keepAliveTime：线程存活时间，当线程数超过核心线程数时，多余的空闲线程的存活时间。</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">unit：时间单位。</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">workQueue：任务队列，用于保存等待执行的任务。</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">threadFactory：线程工厂，用于创建新线程。</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">handler：拒绝策略，当任务无法执行时的处理策略。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">线程池的核心流程图如下：</font>

![1728356434439-eb330bfd-1955-48f7-97ae-f57e0d83efeb.webp](./img/zFpPJ-zmOaU5rTqr/1728356434439-eb330bfd-1955-48f7-97ae-f57e0d83efeb-200612.webp)

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">线程池的工作过程如下：</font>

1. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">线程池初始化：根据corePoolSize初始化核心线程。</font>
2. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">任务提交：当任务提交到线程池时，根据当前线程数判断：</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">若当前线程数小于corePoolSize，创建新的线程执行任务。</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">若当前线程数大于或等于corePoolSize，任务被加入workQueue队列。</font>
1. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">任务处理：当有空闲线程时，从workQueue中取出任务执行。</font>
2. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">线程扩展：若队列已满且当前线程数小于maximumPoolSize，创建新的线程处理任务。</font>
3. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">线程回收：当线程空闲时间超过keepAliveTime，多余的线程会被回收，直到线程数不超过corePoolSize。</font>
4. <font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">拒绝策略：若队列已满且当前线程数达到maximumPoolSize，则根据拒绝策略处理新任务。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">说白了在线程池中，多余的任务会被放到workQueue任务队列中。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">这个任务队列的数据保存在内存中。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">这样就会出现一些问题。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">接下来，看看线程池有哪些问题。</font>

## **<font style="color:rgb(34, 34, 34);background-color:rgb(248, 246, 244);">3 线程池有哪些问题？</font>**
<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">在JDK中为了方便大家创建线程池，专门提供了Executors这个工具类。</font>

### **<font style="color:rgb(34, 34, 34);background-color:rgb(248, 246, 244);">3.1 队列过大</font>**
<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">Executors.newFixedThreadPool，它可以创建固定线程数量的线程池，任务队列使用的是LinkedBlockingQueue，默认最大容量是Integer.MAX_VALUE。</font>

```cpp
public static ExecutorService newFixedThreadPool(int nThreads, ThreadFactory threadFactory) {
    return new ThreadPoolExecutor(nThreads, 
                               nThreads,
                                     0L, 
                  TimeUnit.MILLISECONDS,
     new LinkedBlockingQueue(),
                          threadFactory);
}
```

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">如果向newFixedThreadPool线程池中提交的任务太多，可能会导致LinkedBlockingQueue非常大，从而出现OOM问题。</font>

### **<font style="color:rgb(34, 34, 34);background-color:rgb(248, 246, 244);">3.2 线程太多</font>**
<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">Executors.newCachedThreadPool，它可以创建可缓冲的线程池，最大线程数量是Integer.MAX_VALUE，任务队列使用的是SynchronousQueue。</font>

```csharp
public static ExecutorService newCachedThreadPool() {
  return new ThreadPoolExecutor(0, 
                Integer.MAX_VALUE,
                               60L, 
                  TimeUnit.SECONDS,
    new SynchronousQueue());
}
```

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">如果向newCachedThreadPool线程池中提交的任务太多，可能会导致创建大量的线程，也会出现OOM问题。</font>

### **<font style="color:rgb(34, 34, 34);background-color:rgb(248, 246, 244);">3.3 数据丢失</font>**
<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">如果线程池在执行过程中，服务突然被重启了，可能会导致线程池中的数据丢失。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">上面的OOM问题，我们在日常开发中，可以通过自定义线程池的方式解决。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">比如创建这样的线程池：</font>

```plain
new ThreadPoolExecutor(8, 
                       10,
                       30L, 
     TimeUnit.MILLISECONDS,
    new ArrayBlockingQueue(300),
            threadFactory);
```

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">自定义了一个最大线程数量和任务队列都在可控范围内线程池。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">这样做基本上不会出现OOM问题。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">但线程池的数据丢失问题，光靠自身的功能很难解决。</font>

## **<font style="color:rgb(34, 34, 34);background-color:rgb(248, 246, 244);">4 如何保证数据不丢失？</font>**
<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">线程池中的数据，是保存到内存中的，一旦遇到服务器重启了，数据就会丢失。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">之前的系统流程是这样的：</font>

![1728356433900-a9c6bcef-23b7-4454-881d-eefb99c6b4ee.webp](./img/zFpPJ-zmOaU5rTqr/1728356433900-a9c6bcef-23b7-4454-881d-eefb99c6b4ee-912641.webp)

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">用户请求过来之后，先处理业务逻辑1，它是系统的核心功能。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">然后再将任务提交到线程池，由它处理业务逻辑2，它是系统的非核心功能。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">但如果线程池在处理的过程中，服务down机了，此时，业务逻辑2的数据就会丢失。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">那么，如何保证数据不丢失呢？</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">答：需要提前做持久化。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">我们优化的系统流程如下：</font>

![1728356433848-683932e3-582d-4b60-8a7a-4022ad2f9f8f.webp](./img/zFpPJ-zmOaU5rTqr/1728356433848-683932e3-582d-4b60-8a7a-4022ad2f9f8f-733447.webp)

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">用户请求过来之后，先处理业务逻辑1，紧接着向DB中写入一条任务数据，状态是：待执行。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">处理业务逻辑1和向DB写任务数据，可以在同一个事务中，方便出现异常时回滚。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">然后有一个专门的定时任务，每个一段时间，按添加时间升序，分页查询状态是待执行的任务。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">最早的任务，最先被查出来。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">然后将查出的任务提交到线程池中，由它处理业务逻辑2。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">处理成功之后，修改任务的待执行状态为：已执行。</font>

<font style="color:rgb(100, 100, 100);background-color:rgb(248, 246, 244);">需要注意的是：业务逻辑2的处理过程，要做幂等性设计，同一个请求允许被执行多次，其结果不会有影响。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">如果此时，线程池在处理的过程中，服务down机了，业务逻辑2的数据会丢失。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">但此时DB中保存了任务的数据，并且丢失那些任务的状态还是：待执行。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">在下一次定时任务周期开始执行时，又会将那些任务数据重新查询出来，重新提交到线程池中。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">业务逻辑2丢失的数据，又自动回来了。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">如果要考虑失败的情况，还需要在任务表中增加一个失败次数字段。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">在定时任务的线程池中执行业务逻辑2失败了，在下定时任务执行时可以自动重试。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">但不可能无限制的一直重试下去。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">当失败超过了一定的次数，可以将任务状态改成：失败。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(248, 246, 244);">这样后续可以人工处理。</font>
